## Problem Statement

Pyay lives in SF and wants to plan weeks (and occasional months) ahead by knowing what's worth attending across theatre, tech meetups (Luma-style), and art / museum / gallery openings. Today there is no single place that aggregates these verticals with any taste filter. SF has more events per week in these categories than any human reads — the bottleneck is not event existence, it is curation.

The product is a **personal weekly digest**, delivered every Sunday morning, that pulls events from a finite known list of sources, has Claude rate each one against a written taste profile, and surfaces 5-10 picks worth Pyay's time. No multi-user, no auth, no UI to maintain.

## What Makes This Cool

1. **The taste profile is plain English, not keyword rules.** A 3-paragraph prose description of what Pyay likes ("I'll go to any 60-90 min Luma talk on AI infra; experimental theatre yes; musicals yes when there's a great cast OR it's a popular show; art openings only when there's a wine reception or a real talk attached") is what an LLM grader can act on. Keyword filters can't.
2. **One-click `add to calendar`** via a `.ics` feed. Picks land directly in the calendar app where decisions actually happen — not in a separate app you forget to open.
3. **The "why this one" annotation.** Each pick has a one-liner from Claude explaining why it matched the profile. Reading-not-scanning is the difference between a digest and a list.
4. **It compounds.** Click-tracking (or even just thumbs-up/down feedback) lets the taste profile improve over time. Week 8 picks better than Week 1.

## Constraints

- **Personal use only.** No multi-user, no auth, no public website. Hardcoded user.
- **Free or near-free to run.** Total monthly cost target: under USD 5/month. LLM cost is pennies/week with prompt caching (~50 events/week × 200 tokens/event); the only paid line is potentially a headed-browser GitHub Action runner if scraping needs it.
- **No app store, no daily UI maintenance.** Sunday morning email + a calendar feed. (Static web page deferred to v2.)
- **SF Bay Area only** for v1. No multi-city scaffolding.
- **Three verticals only:** theatre / live performance / comedy; tech / Luma-style meetups; art / museum / gallery openings. No music or nightlife.
- **Digest window:** Sunday 9am PT email covers events starting Sun (today, evening only) through next Saturday end-of-day, in `America/Los_Angeles`. A separate "block now" section covers up to 4 weeks out for events that fill up early.
- **Timezone:** All event timestamps stored and displayed in `America/Los_Angeles`. The `.ics` feed pins `TZID=America/Los_Angeles` with a VTIMEZONE block.

## Premises (all agreed)

1. The bottleneck is curation, not event existence.
2. The source list is finite (~10-15 sources cover 95% of relevant events).
3. Sunday-morning weekly cadence is the right rhythm.
4. Solve YOUR pain only — no audience, no scale concerns yet.
5. The deliverable is a "boring" weekly artifact (email + calendar feed), not an app to open.
6. Distribution is a tiny cron job (GitHub Action or similar).

## Approaches Considered

### Approach A: Bare minimum — keyword filter
Hardcoded sources, fetch weekly, filter by keyword/category, send email. Effort: human ~1-2 days / CC ~1 hour. Risk: low. Curation quality is mediocre because keyword matching can't distinguish "AI panel" from "AI in panel discussions about salmon farming."

### Approach B: LLM-graded picks (RECOMMENDED)
Same aggregation as A, but Claude reads each event description and rates it against a rich taste profile. Output is email + static web page + `.ics` calendar feed. Effort: human ~3-5 days / CC ~3-4 hours. Risk: medium, mitigated by prompt caching keeping cost trivial.

### Approach C: Curator with a voice
Claude becomes the editor — picks 5-7 events and writes the digest as a tiny editorial with opinions and takes. Effort: human ~3-4 days / CC ~3 hours. Risk: high variance — voice can feel like LLM slop if the prompt isn't sharp. Better as a layer on top of B once B is working.

### Stack

- **Language:** TypeScript (familiar, fast iteration, gstack toolchain compatible)
- **Runtime:** Bun (fast cold start matters for cron)
- **Aggregation:** mostly scraping. **Eventbrite's public Search API was deprecated in 2020 — only org-owned event listings are accessible via API**, so Eventbrite's discovery surface needs scraping like the rest. Luma's discovery is JS-rendered and rate-limits; budget headed-browser scraping there too. Theatre venues (ACT, Berkeley Rep, SF Playhouse, Magic) and gallery sites all require custom HTML parsers — one per source. Use `gstack browse` for headed scrapes that need JS execution; plain `fetch` + `cheerio` (or DOM parsing) for static HTML.
- **Per-source parser strategy (CEO Expansion #2 — codify via gstack):**
  - For each source: run `/scrape "<intent describing what to extract>"` to discover and prototype the flow via `$B` primitives
  - When the flow returns the right JSON, run `/skillify` → writes a permanent browser-skill with `script.ts`, `script.test.ts`, and a fixture
  - The wrapper at `sources/sf/<name>.ts` imports and re-exports the skillified script's `run()` function — wrapper is the seam, skillified script is the implementation
  - Subsequent cron runs route to the codified script in ~200ms instead of re-driving the page
  - Path-by-path:
    - **API/RSS:** TodaysTix listings API; some venues may have RSS or iCal — try first, no skillify needed if it's a clean fetch
    - **Static HTML:** still use `/scrape` + `/skillify` for consistency — the auto-generated test catches markup changes
    - **Headed-browser (Luma, Eventbrite, gallery sites):** `/scrape` discovers the JS-rendered page, `/skillify` codifies the flow
- **Source failure handling:** every source wrapped in `try/catch`. A failed source logs to `failed-sources.log`, appears as a footer line in the digest ("⚠️ couldn't fetch: ACT, Magic Theatre"), and never blocks the rest of the run.
- **LLM:** Anthropic SDK with prompt caching. Cache the taste profile + system prompt (5-min TTL); batch events through the cached context in a tight loop so the cache stays warm across all ~50 events in one run. **Grader output schema:** structured JSON per event — `{event_url, rating: 1-10, why: "one sentence", block_now: bool}`. Use Anthropic's tool-use mode for guaranteed JSON parsing. Cost: pennies/week with cache hit rate >90%.
- **Grader failure mode (CEO E1 — degraded mode):** wrap the grader in retry-2x-with-30s-backoff. If still failing after retries, fall through to **degraded email mode**: skip ranking, send all in-window events sorted by source as a "DIGEST DEGRADED — events fetched but not graded, here's the raw list for this week" email. Better than silent failure; you still see what's happening. Implementation: the email template branches on a `degraded: bool` flag set by the grader wrapper.
- **Storage (v1):** single `events.json` file in the repo — array of `{url, title, normalized_key, ics_uid, first_seen_iso, end_iso, rated, rating, why}`. Used to dedupe seen events across weeks. Committed back to the repo at the end of each cron run by the GitHub Action via a `git pull --rebase && git push` to handle the rare manual-commit collision. SQLite + click-feedback deferred to v2.
- **Garbage collection:** end of each run: (1) drop entries from `events.json` where `end_iso < now - 30 days`; (2) drop snapshot directories under `snapshots/` older than 12 weeks.
- **Snapshots (CEO Expansion #3):** every successful cron run also writes `snapshots/YYYY-MM-DD/events.json` (a copy of `events.json` at run time). `--replay YYYY-MM-DD` loads that snapshot instead of running scrapers, regenerates email HTML to stdout (matches `--dry-run`), does not commit. Used to iterate `taste.md` between weekly Sunday runs.
- **Dedupe key:** primary by URL; secondary fuzzy key = `lowercased_title + start_iso_truncated_to_hour + venue_normalized`. **Venue normalization:** lowercase, strip leading "the ", strip punctuation, expand a small hand-maintained alias map (`{"sfmoma": "san francisco museum of modern art", "act": "american conservatory theater", ...}`) which lives in `venues.json`. Add aliases as you find collisions.
- **`.ics` UID stability:** every event gets `ics_uid = sha1(normalized_key) + "@sf-week.local"`, computed once and stored in `events.json`. UIDs persist across weekly runs so calendar subscribers get updates, not duplicates. If an event's `start` changes, the UID stays the same and the calendar entry shifts in place.
- **Dates with ambiguous times** (gallery openings: "April 28, evening"): the grader is given a rule — if no parseable time, assume the canonical time for that vertical (theatre 7:30pm, tech 6:30pm, art opening 6pm). The `.ics` event includes a `DESCRIPTION` line noting the time was estimated. Better than dropping events entirely.
- **Email:** Resend (3,000 free emails/month, more than enough)
- **Calendar feed (E1.1 Pattern A):** `.ics` written to `docs/calendar.ics` by `outputs/ics.ts`. GitHub Pages source set to `/docs` on `main`. Cron commits the regenerated `.ics` back to main alongside `events.json` (single weekly commit: `weekly digest YYYY-MM-DD — N picks, M block-now`). Subscribers hit `https://pyayaungsan.github.io/sf-week/calendar.ics`, never touch again. Having `.ics` in git history is a feature: you can `git diff` week-over-week.
- **Cron:** GitHub Actions, schedule `0 17 * * 0` (UTC) which is **8am PST in winter, 10am PDT in summer** — cron is run in UTC and Actions has no DST awareness. Acceptable: the digest lands somewhere between 8am and 10am Sunday, which is fine. If you want a tighter window, run two crons (`0 17 * * 0` Nov-Mar and `0 16 * * 0` Mar-Nov) and gate them by month inside the workflow. Cron is also best-effort with 5-30 min drift.

### Taste profile (`taste.md` in the repo, version-controlled)

```markdown
# Pyay's taste profile

I live in SF. Weekly digest is for me only — be opinionated, not exhaustive.

## Theatre / live performance
- LOVE: experimental, devised, immersive, 60-90 min, small venues, evening
- LIKE: classic plays at ACT or Berkeley Rep if cast is strong
- LIKE: musicals when (a) there's a great cast in town OR (b) it's a popular show I'd actually want to see — touring Broadway runs at Orpheum / Golden Gate / BroadwaySF count for both. Named leads, recent Broadway openings, or shows everyone's been talking about (Hamilton-tier hype, recent Tony winners, sold-out Broadway transfers)
- SKIP: sketch comedy unless it's a known headliner, anything 3+ hours, regional musical productions that are neither popular shows nor have a notable cast

## Tech / Luma-style meetups
- LOVE: AI infra, agents, eval frameworks, anything where the speaker is actually building
- LIKE: founder fireside if person has shipped something I respect
- SKIP: networking-only, web3, "fireside chat" with no demo, recruiting events
- Weeknight talks ≤ 90 min, Tuesday-Thursday preferred

## Art / museum / gallery
- LOVE: openings with the artist present, panels with curators, retrospectives
- LIKE: Helmut Newton, Wolfgang Tillmans, anything where the photography or video is the medium
- SKIP: fundraiser galas, anything with a $$ ticket I wouldn't pay
```

This file is the model. To improve picks, edit it. No prompt engineering required.

### Output formats (v1)

1. **Email digest** (sent Sunday 9am PT) — top 7 picks for the upcoming week + a "block now" subsection of 1-3 events 2-4 weeks out that need early commits. Each pick: title, date/time (PT), venue, one-line why-it-matched, link to event.
2. **Calendar feed** (`.ics`, subscribed) — picks auto-appear as tentative events in Apple/Google Calendar; accept or delete. VTIMEZONE pinned to `America/Los_Angeles`.

v2 (deferred until needed): static web page with the full ranked list and a "skipped this week" section showing what almost made the cut.

### Slot allocation

The grader returns a global ranking 1-10 across all events. The picker then selects:
- **Up to 9 slots** for the upcoming week, capped at max 3 per vertical (3 tech + 3 theatre + 3 art = 9 ceiling). All picks must score ≥6/10 — no padding with weak picks. If only 4 events score ≥6, the digest has 4 picks. Events scoring <6 are never shown.
- **1-3 slots for "block now"** (>7 days out, up to 4 weeks). The grader is asked, on each candidate, to flag whether it has a capacity-risk reason to commit early — Claude infers this from the event description (e.g. "limited engagement," "tickets on sale Friday," "touring," named speaker the grader recognizes from the taste profile). The signal is a boolean from the grader's structured output, not a separate column.
- **Quiet-week behavior:** if fewer than 3 picks clear the ≥6/10 floor, the email leads with "quiet week — here are the N I'd consider, otherwise skip" and includes a short ranked list of the next 5 events that almost cleared, in case the grader was wrong.

### Why this serves the premise

The taste profile in plain English does what 50 keyword rules can't (premise 1). The fixed source list keeps scope finite (premise 2). Weekly cron, Sunday morning, no UI to open (premises 3+5). Hardcoded user, no auth (premise 4). GitHub Action cron, free hosting (premise 6).

## Success Criteria

- **Week 4:** You've actually read the digest 3 of 4 Sundays.
- **Week 8:** At least one pick per week made it from the digest into your real calendar (accepted via .ics).
- **Week 12:** You've gone to at least 2 events you wouldn't have found without the digest.
- **Anti-success:** You unsubscribe from your own newsletter within a month. That's a clean signal the picks aren't good enough — fix the taste profile or rethink the verticals.

## Reviewer Concerns (consciously deferred)

These came up in adversarial review and are accepted as v2 work or minor-risk:

1. **Email-send failure handling.** If Resend rejects a send, the run logs the error and the digest is missed for that week. No retry, no DLQ. Acceptable for a personal tool — if Resend is down, you'll notice the missing email and re-run manually.
2. **`gstack browse` setup on GitHub Actions runners.** Playwright + Chromium install adds ~90s per run. The workflow setup step uses `bun add @playwright/test && bunx playwright install chromium` with the standard Chromium dependency installation. Assumed implementer-level detail.
3. **Prompt cache hit rate** depends on grading 50 events through a cached prefix in <5 min wall time. Tight Promise.all batching in the grader keeps the cache warm. If wall time drifts above 5 min the cache rebuilds — still pennies/week, just slightly more.
4. **Source markup brittleness.** Each scraper is a custom parser that breaks silently when a site updates. The failure log + digest footer ("⚠️ couldn't fetch: X, Y") makes the breakage visible. Plan to fix one parser per week as needed.
