## Vision

The base design doc (from /office-hours) framed this as "personal SF event aggregator." This CEO review reframes it as **a portable taste model with one initial application (SF events)**. The taste profile in plain English — built from years of going to events and knowing what you skip vs. what you bookmark — is the irreplaceable asset. Scrapers are commodity infrastructure. Designing the project so the taste model is independent of the SF-source layer makes future expansions (travel mode, conference recommendations, friend digests) reachable as features, not rewrites.

The 12-month ideal isn't "the same digest, running reliably" — though that's the floor. It's: same `taste.md`, applied to whatever events matter wherever you are. NYC for a week, a conference schedule someone pastes in, a friend's digest with their own taste profile.

## Accepted Scope (added to this plan)

The following are now part of v1:

- **Directory layout:**
  ```
  sf-week/
  ├── core/
  │   ├── grader.ts        # takes (taste, events) → ranked picks
  │   └── prompt.ts        # cached system prompt
  ├── users/
  │   └── pyay/
  │       └── taste.md           # slot caps stay in core/ for v1; per-user override is v2
  ├── sources/
  │   └── sf/
  │       ├── luma.ts
  │       ├── act.ts
  │       └── ...
  └── outputs/
      ├── email.ts
      └── ics.ts
  ```
- **Per-source parser workflow:** run `/scrape` to discover the flow, then `/skillify` to codify it as a permanent browser-skill with auto-generated `script.ts`, `script.test.ts`, and fixture. Subsequent runs route to the codified script in ~200ms.
- **Replay mode:** `bun run digest --replay 2026-04-21 --user pyay` re-runs the grader+output pipeline against `events.json` snapshot from a past week. No scrapers fire.
- **CLI shape:** `bun run digest --user pyay [--replay YYYY-MM-DD] [--dry-run]`. Future expansion: `--user other` swaps users; `--source-set sf|nyc|<conf>` swaps source bundles.

## Implementation Notes (apply at scaffold time)

These are NOT optional choices — they're the implementation shape implied by the accepted scope. The implementer MUST apply them on Day 1 of the build, not retrofit later.

- **Apply Expansion #1 layout at scaffold time.** Day 1 (currently "create the repo, draft `taste.md`, write `sources.json`") MUST also create `core/`, `sources/sf/`, `users/pyay/` directories. Retrofitting this layout after parsers exist costs 4-6x more.
- **Skillify integration:** `sources/sf/luma.ts` (and each peer) is a thin wrapper that imports and re-exports the skillified script's `run()` function. The parser file is the seam; the skillified script is the implementation. Source files have a stable interface; underlying scripts are regenerated freely via `/skillify`.
- **Replay snapshot semantics:** every successful cron run writes `snapshots/YYYY-MM-DD/events.json` (a copy of `events.json` at run time). Replay loads that snapshot instead of running scrapers fresh. Replay output goes to stdout HTML by default (matches `--dry-run` behavior); `--send` flag would re-send for real (rarely useful, kept for completeness). Snapshots are GC'd at 12 weeks (3x the rolling events.json window).
- **CLI defaults:** `--user` defaults to `pyay` (single hardcoded user in v1). `--source-set` is non-functional in v1; v1 always loads `sources/sf/`. Both flags exist to reserve the namespace.
